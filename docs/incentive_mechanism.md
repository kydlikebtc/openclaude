# OpenClade 子网激励机制设计草案

**版本：** v0.1
**状态：** 草案待审核
**更新：** 2026-03-15

---

## 一、设计目标

OpenClade 激励机制需要同时满足以下目标：

| 目标 | 说明 |
|------|------|
| **服务质量** | 激励 Miner 提供高可用、低延迟的 API 服务 |
| **真实贡献** | 区分真实服务贡献与"待命"贡献 |
| **防欺诈** | 防止 Miner 通过伪造响应或降级模型来刷分 |
| **推广增长** | 激励 Miner 引入真实用户（飞轮效应） |
| **公平分配** | 防止少数 Miner 垄断全部 Emission |
| **冷启动** | 早期矿工较少时仍能维持网络运转 |

---

## 二、TAO Emission 分配结构

```
子网日 Emission
├── Owner      18%  — OpenClade 运营方（子网注册者）
├── Miners     41%  — 提供 API 服务的矿工节点
└── Validators 41%  — 验证服务质量的验证节点
```

> **注：** Bittensor 网络的总 Emission 每年减半，子网在总 Emission 中的占比由网络活跃度动态决定。

## 三、矿工（Miner）评分系统

### 3.1 准入门槛

**全部满足才能参与评分，任一不满足则得 0 分：**

| 指标 | 最低要求 | 说明 |
|------|----------|------|
| 探测成功率 | ≥ 90% | Validator 发送标准探针，Miner 必须正确响应 |
| 在线率 | ≥ 80% | 过去 24 小时内的在线时长占比 |
| 平均延迟 | ≤ 3,000ms | 端到端请求延迟（含 Claude API 调用） |
| 最低质押 | ≥ 5 TAO | 防止低成本注册的垃圾节点 |

### 3.2 最终得分公式

```
最终得分 = 服务分 × (1 + 推荐加成)
```

### 3.3 服务分计算

**服务分 = 贡献分 × W1 + 待命分 × W2**

#### 贡献分（Contribution Score）

```
贡献分 = (Miner 本期处理的 Token 总量 / 全网 Token 总量) × 100
贡献分上限 = 100
```

反映 Miner 对网络的实际贡献量。Token 越多，贡献分越高，但有上限防止垄断。

#### 待命分（Standby Score）

```
待命分 = 在线率 × 50 + 探测成功率 × 50
```

反映 Miner 的可靠性，即使没有实际流量也能获得基础收益。

#### 动态权重（W1, W2）

权重根据网络实际使用量动态调整：

| 网络使用量（日 Token） | W1（贡献权重） | W2（待命权重） | 说明 |
|------------------------|---------------|---------------|------|
| < 10M | 30% | 70% | 早期流量少，重点奖励在线 |
| 10M - 500M | 线性插值 | 线性插值 | 成长期平衡 |
| > 500M | 80% | 20% | 成熟期重点奖励实际贡献 |

**线性插值公式：**
```
usage_ratio = (daily_tokens - 10M) / (500M - 10M)
W1 = 30% + usage_ratio × (80% - 30%)
W2 = 1 - W1
```

### 3.4 推荐加成（Referral Bonus）

```
推荐加成 = min(当期推荐用户消费额 / 全网消费额 × 2 × 30%, 30%)
```

**规则：**
- 只统计**本计费周期**内的推荐用户消费（不积累历史）
- 只统计**当期消费 ≥ $5** 的有效推荐用户（防刷量）
- 推荐加成最高 **30%**（即服务分翻倍效果有限）
- 推荐关系通过 OpenClade 平台专属链接绑定，链上可验证

**示例：**

| 场景 | 推荐用户消费 | 全网消费 | 推荐占比 | 推荐加成 |
|------|------------|---------|---------|---------|
| 无推荐用户 | $0 | $10,000 | 0% | 0% |
| 小推荐 | $500 | $10,000 | 5% | min(5%×2×30%, 30%) = 3% |
| 大推荐 | $2,500 | $10,000 | 25% | min(25%×2×30%, 30%) = 15% |
| 超级推荐 | $6,000 | $10,000 | 60% | min(60%×2×30%, 30%) = 30% |

### 3.5 评分平滑

为防止单次测试抖动影响收益，采用**指数移动平均（EMA）**平滑评分：

```
平滑得分 = 0.7 × 当期原始得分 + 0.3 × 上期平滑得分
```

## 四、验证者（Validator）评分系统

验证者通过**定期抽样探测**来评估 Miner 质量。

### 4.1 验证频率

| 阶段 | 验证间隔 | 每次探测 Miner 数 |
|------|---------|-----------------|
| 早期（< 20 Miner） | 每 5 分钟 | 全量 |
| 成长期（20-100 Miner） | 每 2 分钟 | 随机 50 个 |
| 成熟期（> 100 Miner） | 每 1 分钟 | 随机 30 个 |

### 4.2 响应质量评估

**综合评分 = 可用性 × 20% + 延迟 × 15% + 响应质量 × 35% + 一致性 × 20% + 效率 × 10%**

#### 可用性分（Availability）

```python
if is_timeout:
    return 0.0
if response is None or len(response) == 0:
    return 0.0
return 1.0
```

#### 延迟分（Latency）

| 延迟 | 得分 |
|------|------|
| < 2秒 | 1.0 |
| 2-5秒 | 0.7 - 1.0（线性） |
| 5-10秒 | 0.3 - 0.7（线性） |
| > 10秒 | 0 - 0.3（线性衰减） |

#### 响应质量分（Quality）

- **长度合理性**：< 10词 或 > 50,000词 扣分
- **错误检测**：包含错误标志词（"Error"、"Exception"）扣分
- **格式正确性**：对于期望 JSON 的响应，验证 JSON 有效性
- **语言连贯性**：简单启发式检查语法质量

#### 一致性分（Consistency）

- 对同一标准测试集，不同 Miner 的响应相似度对比
- 与历史基准响应（由 Validator 本地持有）的相似度对比
- 防止 Miner 返回与 Claude 风格不一致的伪造响应

#### 效率分（Efficiency）

```
效率分 = min(基准Token数 / 实际使用Token数, 1.0)
基准Token数 = 500（Claude Sonnet 的信息密度参考值）
```

### 4.3 模型降级检测

Validator 内置**模型质量基准**，检测 Miner 是否用低端模型假冒高端模型：

| 声称模型 | 最低质量分 |
|---------|-----------|
| claude-opus-4 | 0.95 |
| claude-sonnet-4 | 0.85 |
| claude-haiku-4 | 0.70 |

若响应质量分低于阈值，判定为**降级欺诈**，该 Miner 本轮得 0 分。

## 五、防欺诈机制

### 5.1 常见作弊方式与对策

| 作弊方式 | 检测方法 | 惩罚 |
|---------|---------|------|
| 模型降级（Haiku 冒充 Opus） | 质量分基准检测 | 本轮 0 分 |
| 硬编码伪造响应 | 一致性检测 + 随机查询 | 本轮 0 分 + 持续监控 |
| 使用失效 API Key | 探测成功率下降 | 低于 90% 则不得分 |
| 刷推荐（虚假用户） | 消费 ≥ $5 过滤 | 推荐加成归零 |
| 女巫攻击（大量假节点） | 质押门槛 + Yuma Consensus | 稀释 Emission |

### 5.2 Yuma Consensus 的内生保护

Bittensor 的 Yuma Consensus 为 Validator 设计了诚实投票的**自我激励**：

- Validator 的 Dividends 取决于其给高分 Miner 的权重
- 若 Validator 给了差 Miner 高分，其 Dividends 会被"诚实"的 Validator 拉低
- 因此，Validator 有经济动机**如实评分**，而非串通 Miner

## 六、冷启动激励

网络初期矿工数量少，采用**冷启动加成**避免空转：

```python
def bootstrap_bonus(num_miners: int, month: int) -> float:
    if month > 3:
        return 0.0  # 3个月后取消
    if num_miners < 10:
        return 0.5  # 前10个矿工额外+50%
    elif num_miners < 20:
        return 0.3  # 10-20个矿工额外+30%
    else:
        return 0.0
```

## 七、Claude API 响应质量的特殊评估挑战

由于 Claude API 的生成有**随机性**（temperature > 0），对同一问题的两次响应不一定相同，评估一致性时需特别处理：

### 7.1 标准化测试题库

Validator 维护一个**标准测试题库**，包含：

- **确定性问题**（数学、代码、事实）：期望高度一致
  - "1 + 1 等于几？" → 期望："2"
  - "Python 中如何反转列表？" → 期望：特定代码片段

- **开放性问题**（创意写作、解释）：允许适当差异
  - 使用语义相似度（而非完全匹配）评估一致性

### 7.2 跨 Miner 相似度

对同一问题查询多个 Miner，计算响应的**余弦相似度**。若某 Miner 的响应与其他 Miner 平均相似度 < 0.7，触发**异常警告**。

## 八、参数配置总览

| 参数 | 默认值 | 可调范围 | 说明 |
|------|--------|---------|------|
| 探测成功率门槛 | 90% | 85-95% | 过低影响网络质量 |
| 在线率门槛 | 80% | 70-90% | 过低允许太多离线 |
| 最大延迟 | 3,000ms | 2,000-5,000ms | 影响用户体验 |
| 最低质押 | 5 TAO | 1-20 TAO | 防刷节点 |
| 推荐加成上限 | 30% | 10-50% | 影响推广激励强度 |
| EMA 平滑因子 | 0.7 | 0.5-0.9 | 0.9 = 更慢更稳定 |
| 模型降级检测质量门槛 | Opus: 0.95 | ± 0.05 | 需要更多实验校准 |
